Regime tells vs traps

Read a kernel's source, guess where it bottlenecks. Sometimes the guess is a tell. Sometimes it's a trap only measurement can spring.

DeviceNVIDIA A100-SXM4-40GB
ComputeCC 8.0 · 108 SMs
Peaks19.5 TFLOP/s · 1555 GB/s
The premise

You can read the source and form a hypothesis.
You cannot trust it.

The tell

Count FLOPs and bytes straight from the source → arithmetic intensity. Place it against the roofline ridge (peak compute ÷ peak BW ≈ 12.5 FLOPs/byte on the A100) for a source-level guess at the regime:

  • HBM-bound — AI below the ridge; device DRAM bandwidth is the ceiling
  • compute-bound — AI above the ridge; SM FP32 units are the ceiling
  • latency-bound — a third regime the 2-axis roofline can't draw: under both roofs, pipeline starved. Predicted from compile stats + occupancy, not AI.
The reality

The real operating point comes from measurement: achieved GFLOP/s, apparent GB/s, occupancy, and the local/regs/smem compile stats. They confirm the tell — or expose a trap.

Thesis  ·  source = hypothesis  →  compile stats + occupancy + a measurement = the verdict
The bench · A100 roofline

Two roofs, a ridge — and a third regime

The ridge is a device constant
ridge = peak compute / peak BW
      = 19.5 TFLOP/s / 1555 GB/s
      ≈ 12.5 FLOPs/byte

below → HBM-bound  ·  above → compute-bound

The third regime · latency-bound

The roofline assumes the pipeline is always fed — enough independent work in flight to hide each op's latency. When it isn't, you saturate neither roof: the SMs sit idle, waiting.

Causes: dependent chains (low ILP), too few warps (low occupancy), uncoalesced access, register spills.  Flagged when max(frac_bw, frac_fl) < 0.25.

Exhibit 1 · the control · solve for the flip

AI-dial — the tell holds ✓

The tell · AI is a function of iters
// 2 FLOPs/iter, 8 B I/O (1 read + 1 write)
for (k = 0; k < iters; k++)
    x = fmaf(x, 1.0001f, 1e-7f);
AI(iters) = (2·iters) / 8  =  iters / 4

iters/4 = 12.5  →  iters ≈ 50   // hits the ridge

iters < 50  →  HBM-bound
iters > 50  →  compute-bound
The reality · verdict flips on schedule
iters = 8 → AI = 2
72% HBM
iters = 64 → AI = 16
compute-bound
flips across iters ≈ 50, exactly as predicted

The twist: the compute end tops out at only ~39% of peak FP32 — a single dependent FMA chain has no ILP, so it's latency-bound, not at the roof. Compute-bound ≠ peak →

Exhibit 1 · two regimes, two knob sets

The roofline says which ceiling. The fix differs completely.

HBM-bound · iters < 50 · limited by bytes

You can't out-compute a memory wall — adding FLOPs is free until the ridge.

  • Move fewer bytes — fp16/bf16 (~2× BW); fuse kernels to kill DRAM round-trips (biggest lever)
  • Move them efficiently — keep loads coalesced (Ex5: 20× swing); vectorize (float4 / 128-bit)
  • Saturate the bus — enough in-flight requests (occupancy / MLP)
  • Or leave the regime — raise AI past the ridge
compute-bound · iters > 50 · limited by latency

The 39% wall: unroll 1 serializes the FMA chain (~4 cyc each), pipeline idles.

  • Increase ILP — split the chain into independent accumulators (headline knob)
  • Increase TLP / occupancy — more warps hide latency (ILP & TLP substitute)
  • Never spill — keep work in registers (Ex4: 13× from one spill); watch local == 0
  • Right instructions — fmaf, avoid div/transcendentals; tensor cores for matmul
Bytes are irrelevant once compute-bound; ILP is irrelevant when HBM-bound. Know the regime before you tune.

Far below the ridge. Two reads, one write, almost no arithmetic. Prediction: HBM-bound, no surprises.

Exhibit 2 · should it even run on the GPU?

The regime says how it runs — not where it should

CPU DRAM
~200 GB/s

where host data lives

PCIe Gen4 ×16
~25 GB/s

the transfer tax

GPU HBM
~1555 GB/s

where device data lives

data already on the GPU → GPU wins

1555 vs 200 GB/s ≈ ~8× faster. No PCIe in the loop — the on-device roof is the only ceiling.

data in CPU RAM → CPU wins

Crossing PCIe at 25 GB/s costs more than the trivial math. For SAXPY this is the classic "not worth offloading" case.

PCIe ridge: 19.5 TFLOP/s ÷ 25 GB/s ≈ 780 FLOPs/byte to amortize the transfer. SAXPY's 0.17 is ~4600× short.

Compute-bound kernels are where the GPU wins decisively — amortize the one-time transfer over a mountain of compute and the PCIe tax vanishes.

Exhibit 3 · ★ star trap

L2-reuse — the source lies about HBM

The tell · "obviously HBM-bound"
// re-reads a 4 MB buffer, reuse×
for (k = 0; k < reuse; k++)
    acc += small[(i + k) % m];
working set = m = 2^20 floats = 4 MB
accesses    = N · reuse   (16M · 64 ≈ 1.07e9)
src bytes   = 4·N·reuse ≈ 4.29 GB   // each access = DRAM read
source AI   = 1 add / 4 B = 0.25 FLOPs/byte → HBM-bound
The reality · physically impossible
apparent BW = src bytes / time
2726 GB/s
= fraction of HBM peak
175%
2726 / 1555 = 1.75  →  can't come from DRAM
4 MB working set ⊂ 40 MB L2  →  real DRAM ≈ 4 MB
over-count = 4.29 GB / 4 MB ≈ 1024× served by L2
TRAP — source says HBM-bound; reality is cache-served. Locality moved the roof.
Exhibit 4 · the most dramatic trap

Register spill — a one-character change, 13× slower

The tell · both look compute-bound
// static index → stays in registers
r[j] = fmaf(r[j], r[(j+1)&15], …);

// runtime index → forced to LOCAL mem
r[j] = fmaf(r[j], r[(j+d)&15], …);
r[16] floats = 16 · 4 B = 64 B / thread
ptxas → local_size_bytes = 64   // compile-time stat

A register file isn't addressable. A runtime index d forces r[] out of registers; ptxas spills it to local memory and reports the size.

The reality · register → memory
static index · operands in registers
18211 GFLOP/s
runtime index · local = 64 B/thread
1362 GFLOP/s

13× slowdown — identical FLOPs.

Local is a logical space — backed by DRAM, cached in L1. Each FMA now waits on a load instead of reading a free register: added latency, not bandwidth, craters it.

TRAP — the disguise stays "compute-bound"; only local_size_bytes > 0 reveals the register→memory spill that reads latency-bound.
Exhibit 5 · the reframe

Uncoalesced — same regime, 20× off the roof

The tell · identical to SAXPY
// scatter consecutive threads
i = (t * 131) % n;
c[i] = 2.0f*a[i] + b[i];
AI = 2 FLOPs / 12 B = 0.17   // same as SAXPY
stride = 131 floats = 524 B between neighbors
addresses don't appear in AI → predicts HBM-bound
The reality · 32-byte sectors
warp = 32 threads; HBM moves 32 B sectors
coalesced: 32 floats = 128 B = 4 full sectors (100%)
strided:   32 threads → 32 sectors, 4 B used each
           = 4/32 = 12.5% useful per fetch
coalesced (SAXPY)
1291 GB/s
strided (scattered)
65 GB/s

~20× collapse — and both still classify HBM-bound, the label does not flip.

Reframe — coalescing decides whether you reach 83% or 4% of the roof. Not a misclassification; a collapse.
Exhibit 6 · the deliberate tell

Shared memory — locality you choose, occupancy you pay

The tell · 32 KB/block staging buffer
__shared__ float tile[8192]; // 32 KB
// stage once, reuse 256× from smem

The chosen cousin of the Ex3 L2 trap: deliberate on-chip reuse. But __shared__ consumes a per-SM resource.

The reality · the binding cap flips
smem per block
32768 B
occupancy ceiling · cap flips thread → smem
5 blocks/SM = 62%

164 KB/SM ÷ 32 KB = 5. The min(thread, block, reg, smem) ceiling now binds on smem.

tell holds — both faces of shared memory: on-chip reuse and its occupancy cost.
Synthesis

Read the source to form a hypothesis.
The compile stats, occupancy, and a measurement are what confirm or destroy it.

Knob 1 · AI — picks the roof  (Ex1, Ex2)
Knob 2 · locality — moves the roof  (Ex3, Ex6)
Knob 3 · access efficiency — decides if you reach it  (Ex4, Ex5)
  • Three regimes, not two. Latency-bound lives under both roofs — the 2-axis roofline can't draw it.
  • Compute-bound ≠ peak. Ex1 tops at 39% — ILP and latency still matter above the ridge.
  • The traps are invisible in the math. Cache reuse, local spills, scatter — none change AI.
  • The tells are in the stats. regs, smem, local, occupancy, apparent GB/s.
